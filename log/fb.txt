

Documentation Migration Plan: Old Project to New Project Structure

## Phase 1: Analysis and Inventory

1. Read and analyze all old project documentation files to extract:
   - Business flows and processes
   - Application pages and components
   - Data models and structures
   - Configuration details
   - Dependencies and integrations
```
docs/migration/old-project/code/application_pages_and_components.md
docs/migration/old-project/code/application_replication_guide.md
docs/migration/old-project/code/business_flows_and_processes.md
docs/migration/old-project/code/configuration_and_environment_setup.md
docs/migration/old-project/code/data_models_and_structures.md
docs/migration/old-project/code/dependencies_and_integrations.md
docs/migration/old-project/code/project_architecture_overview.md
```

2. Review new project documentation structure to understand:
   - Current documentation organization
   - Existing features and capabilities
   - Technology stack and architecture patterns
   - Documentation standards and formats
```
docs/migration/new-project/api-documentation.md
docs/migration/new-project/application-replication-guide.md
docs/migration/new-project/configuration-and-environment-setup.md
docs/migration/new-project/data-models-and-structures.md
docs/migration/new-project/dependencies-and-integrations.md
docs/migration/new-project/deployment-guide.md
docs/migration/new-project/performance-optimization.md
docs/migration/new-project/project-architecture-overview.md
docs/migration/new-project/README.md
docs/migration/new-project/security-best-practices.md
```

3. Create a content mapping matrix:
   - List all old project features/pages/flows
   - Identify corresponding sections in new project structure
   - Mark gaps where old project content needs to be added
   - Note conflicts or overlaps that need resolution

## Phase 2: Content Extraction and Organization

4. Extract business flow documentation:
   - Copy all business processes from old project
   - Adapt to new project's terminology and context
   - Update with any new technologies or patterns from new project

5. Extract application pages and components:
   - Document all UI pages from old project
   - Map to new project's component architecture
   - Note any redesign or modernization needed

6. Merge data models:
   - Compare old and new data structures
   - Create unified data model documentation
   - Highlight migration paths or transformations

7. Consolidate configuration and setup:
   - Combine environment setup guides
   - Prioritize new project's approach
   - Include old project specific configurations as needed

## Phase 3: Documentation Creation

8. Create final documentation structure:
   ```
   docs/final-project/
   ├── README.md
   ├── project-architecture-overview.md
   ├── business-flows-and-processes.md
   ├── application-pages-and-components.md
   ├── data-models-and-structures.md
   ├── configuration-and-environment-setup.md
   ├── dependencies-and-integrations.md
   ├── api-documentation.md
   ├── deployment-guide.md
   ├── performance-optimization.md
   └── security-best-practices.md
   ```

9. Write or update each documentation file:
   - Start with new project's template/structure
   - Incorporate all relevant old project content
   - Ensure consistency in terminology and formatting
   - Add cross-references between related documents

10. Review and validate:
    - Check for completeness against both projects
    - Verify all business flows are documented
    - Ensure technical accuracy with new project's stack
    - Validate that documentation follows established standards

## Phase 4: Finalization

11. Create documentation index:
    - Add comprehensive table of contents
    - Include quick reference sections
    - Add navigation aids

Tôi cần bạn so sánh và lấy toàn bộ tài liệu old project (chủ yếu là các business flow, tính năng, pages) và kết hợp với cấu trúc thiết kế mới trong tài liệu của new project.
đẻ tạo ra một folder tài liệu final, đảm bảo chứa đấy đủ các features, pages, business flow của old project nhưng theo cấu trúc của new project.
tài liệu đó sẽ là tài liệu chính thống cho toàn bộ project, nên cần chỉnh chu, đầy đủ và chi tiết.

Lưu ý:
- Sử dụng mode Documentation Specialist để viết các file md nhé.
- các subtask cần đủ nhỏ để có thể dễ dàng hoàn thành theo checklist
- ưu tiên sử dụng các tech mới hơn từ new project.


STRICT CODING RULES:

Modularization: Break logic, components, master file into small, single-responsibility modules, child components, child files. Avoid monolithic files.

No Duplication: Always search existing files/codebase before creating new ones.

Verbose Naming: Use highly descriptive filenames (e.g., user_authentication_service.ts instead of auth.ts) so the filename reveals the content without opening it (optimizes grep/glob search).
